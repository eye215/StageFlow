# StageFlow 제품·기능 명세서

문서 버전: 1.0  
기준 앱: GitHub Pages v148  
목적: 기능 추가의 기준점을 만들고, 전반적인 UI/UX 및 데이터 구조 개편 범위를 합의한다.

## 1. 제품 정의

StageFlow는 뮤지컬·연극 공연팀이 대본과 운영표를 장면 중심 데이터로 정리하고, 연습부터 공연 당일까지 같은 정보를 공유하는 모바일 우선 공연 운영 도구다.

핵심 가치는 다음 세 문장으로 정리한다.

1. 흩어진 대본·표·음악·악보를 장면별 운영 정보로 바꾼다.
2. 배우·배역·의상·소품·큐를 서로 연결해 각 팀원에게 필요한 정보만 보여준다.
3. 공연·리허설 중에는 GO를 기준으로 시간과 준비 상태를 실시간 공유한다.

## 2. 핵심 사용자

| 사용자 | 주요 목적 | 기본 화면 |
| --- | --- | --- |
| 제작·관리자 | 공연 생성, 팀 초대, 자료 정리, 권한 관리 | 공연 준비 현황 |
| 연출·조연출 | 장면·배역·연습·피드백 관리 | 장면 및 할 일 |
| 무대감독 | 큐, 소품·대도구, GO, 런타임 관리 | 공연·리허설 모드 |
| 의상·소품팀 | 장면별 준비물과 IN/OUT 확인 | 담당 업무 |
| 음향·음악팀 | 넘버별 복수 음악 파일과 큐 관리 | 음악·큐 |
| 배우 | 내 장면, 배역, 의상, 소품, 다음 등장 확인 | 개인 브리핑 |

## 3. 제품 구조 원칙

- 공연이 최상위 작업 단위다.
- 장면이 모든 운영 정보의 중심이다.
- 배우와 배역은 분리하되, 한 배우가 여러 배역을 가질 수 있다.
- 같은 정보는 한 곳에서만 편집하고 다른 화면에서는 연결해 보여준다.
- 자동 분석 결과는 바로 저장하지 않고 미리보기·선택·수정 후 적용한다.
- 자동화가 확실하지 않으면 기존 데이터를 유지하고 확인 필요 상태를 표시한다.
- 공연 중 화면은 편집보다 읽기, 큰 버튼, 즉시 상태 확인을 우선한다.

## 6. 현재 구조의 주요 문제

### P0: 데이터 신뢰성과 동시 편집

- 장면은 DB 테이블, 배우·소품·운영 상태는 Storage JSON에 섞여 있다.
- JSON 전체 덮어쓰기는 다수 팀원의 동시 편집에 취약하다.
- 배우·배역·장면·소품 관계가 문자열에도 중복 저장된다.
- 팀과 공연의 권한 경계가 UI와 RLS 양쪽에서 명확히 검증되어야 한다.

### P1: 화면 및 탐색 중복

- 공연 내부 메뉴가 많고 `더보기`의 역할이 불명확하다.
- 개요, 홈, 준비도, 일정, 할 일에서 비슷한 정보가 반복된다.
- 자동정리 화면이 업로드, 분석, 검토, 적용, 자료실을 한 화면에 모두 담는다.
- 배우 카드 내부에 편집, 장면, 준비물, 공유 기능이 과도하게 모여 있다.

### P1: 코드 구조

- 핵심 UI와 비즈니스 로직 대부분이 `src/App.jsx` 한 파일에 집중되어 있다.
- 기능별 CSS 파일이 누적되어 우선순위 충돌 가능성이 높다.
- 자동 분석 파서, 저장 로직, 화면 상태가 강하게 결합되어 테스트가 어렵다.
- 클라이언트가 Storage 파일 경로와 병합 규칙을 직접 관리한다.

### P2: 자동 분석 품질

- 스캔 PDF는 OCR이 없어 텍스트를 추출할 수 없다.
- 표의 병합 셀과 다단 레이아웃은 규칙 분석만으로 오인식될 수 있다.
- 배우명, 실제 배우, 배역명이 원본 표마다 다르게 쓰여 표준화가 필요하다.
- 자동 생성 결과의 출처와 신뢰도 기록이 부족하다.

## 7. 목표 데이터 모델

JSON 파일 중심 저장을 아래 관계형 테이블로 단계 이전한다.

| 영역 | 주요 엔터티 |
| --- | --- |
| 조직 | workspaces, workspace_members |
| 공연 | productions, production_members, invitations |
| 구성 | acts, scenes, numbers |
| 인물 | people, roles, cast_assignments, scene_cast |
| 준비물 | costumes, costume_changes, props, prop_movements, set_pieces |
| 진행 | cues, cue_runs, show_sessions, scene_runs, readiness_states |
| 자료 | source_files, media_files, file_links, import_jobs |
| 업무 | tasks, schedules, attendance, feedback |

필수 관계:

- `person 1:N cast_assignments`
- `role 1:N cast_assignments`
- `scene N:M cast_assignments`
- `scene 1:N cues`
- `scene 1:N prop_movements`
- `number 1:N media_files`
- `import_job 1:N import_candidates`

## 8. 목표 자동정리 흐름

1. 자료 업로드
2. 파일 유형 판별
3. PDF 텍스트·표 추출 또는 OCR
4. 헤더·열 의미 추론
5. 기존 장면·배우·배역·넘버와 후보 매칭
6. 신규·변경·중복·충돌 구분
7. 적용 대상과 저장 방식 선택
8. 행별 수정 및 제외
9. 자동 백업
10. 트랜잭션 적용
11. 변경 이력과 원본 출처 저장

모든 자동 생성 항목은 `source_file_id`, `import_job_id`, `confidence`, `review_status`를 가져야 한다.

## 9. UI/UX 개편 명세

### 홈

- 기본 공연 한 개에 집중한다.
- 오늘 일정, 내 할 일, 다음 연습, 준비 경고만 표시한다.
- 공연 전체 통계와 중복되는 카드 제거.

### 공연 개요

- 전체 준비도 대신 `확인이 필요한 항목`을 우선한다.
- 장면, 배우, 준비물, 큐의 미완료 수를 행동 버튼으로 제공한다.
- 최근 변경 내역을 표시한다.

### 자동정리

- 1단계 업로드 → 2단계 분석 → 3단계 검토 → 4단계 적용으로 화면을 분리한다.
- 검토 화면은 표 형태와 모바일 카드 형태를 모두 지원한다.
- 신규, 변경, 충돌, 제외 상태를 색과 텍스트로 함께 표시한다.
- 적용 후 결과 보고서와 되돌리기를 같은 화면에서 제공한다.

### 배우

- 1차 단위는 배우, 2차 단위는 배역이다.
- 배우 카드 상단에는 이름, 전체 배역, 참여 팀원, 등장 장면 수만 표시한다.
- 상세 진입 후 장면, 의상, 소품, 피드백을 탭으로 분리한다.
- 수평 장면 스크롤은 사용하지 않고 접힌 세로 목록 또는 장면 번호 요약을 사용한다.

### 공연 모드

- 상단: 현재 장면, 경과 시간, CONTROL/FOLLOW 상태.
- 중앙: 다음 GO 큐 한 개를 가장 크게 표시.
- 하단: 배우 준비, 의상, 소품, 음악의 문제 항목만 표시.
- 편집 버튼은 숨기고 상태 체크와 GO만 강조한다.
- 개인 모드는 내 다음 등장과 준비물에 집중한다.

## 10. 비기능 요구사항

- 모바일 360px 이상에서 수평 페이지 스크롤이 없어야 한다.
- 공연 모드의 핵심 버튼은 최소 48px 터치 영역을 갖는다.
- 오프라인에서 마지막 공연 데이터와 음악 목록을 읽을 수 있어야 한다.
- 저장 성공, 저장 중, 오프라인, 충돌 상태를 구분해 표시한다.
- 공연 중 동기화는 2초 이내, 준비 화면은 10초 이내 갱신을 목표로 한다.
- 모든 삭제·대량 변경은 확인, 백업, 되돌리기 중 두 가지 이상을 제공한다.
- 업로드 파일은 비공개가 기본이며 공연 참여자만 접근한다.
- 비밀키는 클라이언트 번들에 포함하지 않는다.

## 11. 개편 로드맵

### Phase 1: 화면 구조 정리

- 메뉴를 `개요·장면·배우·준비·공연·더보기`로 축소
- 홈과 공연 개요의 중복 제거
- 자동정리 4단계 화면 분리
- 배우 상세 화면 분리

### Phase 2: 데이터 구조 이전

- people, roles, cast_assignments, scene_cast 테이블 도입
- props, cues, readiness_states 테이블 이전
- JSON 저장은 마이그레이션 기간 읽기 호환만 유지
- 변경 이력과 충돌 방지 도입

### Phase 3: 공연 운영 안정화

- 실시간 채널 기반 GO·HOLD·준비 상태
- 연결 끊김 및 재접속 처리
- 권한별 CONTROL 잠금
- 공연 세션 리포트

### Phase 4: 분석 고도화

- OCR
- 표 헤더 자동 매핑 편집기
- 장면·배역·넘버 후보의 신뢰도 표시
- 음악·악보·대본 교차 연결

## 12. 다음 수정 전에 결정할 항목

1. 앱의 1차 고객이 배우 개인인지, 무대감독·운영팀인지.
2. 배우 실명과 배역명을 누가 관리하는지.
3. 공연별 한 사용자가 한 배우만 선택하는지, 여러 배우를 선택할 수 있는지.
4. 자동정리의 기본 저장 방식이 신규 추가인지 기존 병합인지.
5. 공연 중 큐 진행 권한을 한 명만 가질지 여러 명이 가질지.
6. 의상과 소품의 준비 완료가 공연 회차마다 초기화되어야 하는지.
7. 무료 버전에서 팀원·파일·공연 수 제한을 둘지.
